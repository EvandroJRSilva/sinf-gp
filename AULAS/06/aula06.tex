\section{Gerenciamento de configuração}

\begin{frame}{Gerenciamento de configuração}
    \begin{itemize}
        \justifying
        \item<1-5,7> Os softwares \textbf{mudam constantemente} durante seu desenvolvimento e uso
            \begin{itemize}
                \justifying
                \item<2-5,7> Defeitos são descobertos.
                \item<3-5,7> Requisitos mudam.
                \item<4-5,7> Novas versões de hardware e software.
                \item<5,7> Concorrentes lançam novidades.
            \end{itemize}
        \item<6-7> À medida em que são feitas alterações no software, uma nova versão de um sistema é criado. A \textbf{maioria dos sistemas}, portanto, pode ser encarada como um \textbf{conjunto de versões}, \textbf{cada uma} delas tendo que ser \textbf{mantida e gerenciada}.
    \end{itemize}
\end{frame}

\begin{frame}{Gerenciamento de configuração}
    \begin{itemize}
        \justifying
        \item O gerenciamento de configuração
            \begin{itemize}
                \justifying
                \item<1,4> Está relacionado às \textbf{políticas}, aos \textbf{processos} e às \textbf{ferramentas} para o gerenciamento de sistemas de software que mudam.
                \item<2,4> É \textbf{útil} para \textbf{projetos individuais}, pois é fácil para uma pessoa esquecer quais alterações foram feitas.
                \item<3-4> É \textbf{essencial} para \textbf{projetos em equipe}, nos quais vários desenvolvedores trabalham em um sistema de software ao mesmo tempo.
            \end{itemize}
    \end{itemize}
\end{frame}

\begin{frame}{Gerenciamento de configuração}
    \begin{itemize}
        \justifying
        \item Quatro atividades principais
            \begin{enumerate}
                \justifying
                \item<1> \textbf{Controle de versão}: envolve manter o controle das várias versões dos componentes do sistema e garantir que as mudanças feitas em componentes por diferentes desenvolvedores não interfiram umas com as outras.
                \item<2> \textbf{Construção de sistema}: processo de reunir componentes, dados e bibliotecas do programa, compilando-os e ligando-os para criar um sistema executável.
                \item<3> \textbf{Gerenciamento de mudanças}: envolve manter o controle das solicitações de mudança de clientes e desenvolvedores no software já entregue, elaborar os custos e o impacto de fazer essas mudanças e decidir se e quando as alterações devem ser implementadas.
                \item<4> \textbf{Gerenciamento de lançamentos} (\textit{releases}): envolve a preparação de software para o lançamento externo e o acompanhamento das versões de sistema que foram lançadas para uso do cliente.
            \end{enumerate}
    \end{itemize}
\end{frame}

\section{Gerenciamento de versões}

\begin{frame}{Gerenciamento de versões}
    \begin{itemize}
        \justifying
        \item O que é controle de versão? Onde vive? O que faz? Do que se alimenta?
        \begin{itemize}
            \justifying
            \item<2-> É um \textbf{sistema} que \textbf{armazena modificações} em um arquivo ou conjunto de arquivos em função do tempo, e \textbf{permite o acesso a versões específicas a qualquer momento}.
            \item<3> Em nosso contexto usamos para versionar códigos-fonte, mas um controle de versionamento pode ser aplicado sobre praticamente qualquer tipo de arquivo.
        \end{itemize}
    \end{itemize}
\end{frame}

\begin{frame}{Gerenciamento de versões}
    \begin{itemize}
        \justifying
        \item Com um \textbf{Sistema de Controle de Versionamento} (VCS) é possível
        \begin{itemize}
            \justifying
            \item<2-> Voltar ao estado anterior dos arquivos selecionados;
            \item<3-> Voltar ao estado anterior de todo o projeto;
            \item<4-> Comparar diferentes modificações;
            \item<5-> Caguetar a pessoa que fez uma modificação que está causando erro;
            \item<6-> Saber quem adicionou alguma nota (sobre problema ou sugestão), onde e quando;
            \item<7-> Se você sem querer apagou tudo, com um VCS é maior a probabilidade de você conseguir recuperar o que perdeu;
            \item<8> Tudo isso sem muito custo computacional.
        \end{itemize}
    \end{itemize}
\end{frame}

\subsection{VCS Distribuído}

\begin{frame}{VCS Distribuído}
    \begin{itemize}
        \justifying
        \item Agora os clientes (programadores) não somente verificam o último estado dos arquivos (como era no centralizado).
        \item Cada um espelha o repositório por completo, incluindo todo o seu histórico.
        \item Portanto, todos os envolvidos naquele projeto são, por si só, backups.
        \item<2> Outros VCS Distribuídos além do Git: Mercurial, Bazaar e Darcs.
    \end{itemize}
\end{frame}
